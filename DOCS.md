# MIRAGE — AI Honeypot Defense Platform

> **Alpha software.** Core detection and deception pipeline is stable. Some features listed here are partially implemented or stubbed — they are marked ⚠️ where relevant.

## Concept

MIRAGE protects AI agents from adversarial attacks. Instead of silently blocking attackers, the system **deceives** them: it replaces the real AI agent with a convincing decoy that responds with fabricated data while the real system stays safe.

How it works:

1. An attacker sends a message to your agent (prompt injection, jailbreak attempt, system-prompt extraction, etc.)
2. **Lobster Trap** analyses the message and assigns a `risk_score` from 0 to 1
3. If the score exceeds the threshold (default 0.6) the session is switched to **honeypot** mode
4. All subsequent responses are generated by a **decoy LLM** — with fake data, pretending to be the real agent
5. The attacker believes they have compromised the system and reveals their techniques — everything is logged

The value: the attacker does not know they are in a trap, and you get a full threat profile without exposing any real data.

## Architecture

```
Attacker ──► POST /chat ──► Backend (Go/Gin :8081)
                                │
                    ┌───────────┴────────────┐
                    ▼                        ▼
            Lobster Trap (:8080)        Redis (:6379)
            (DPI analysis)             (storage)
                    │
                    ▼
            risk_score ≥ 0.6?
                    │
          ┌─────────┴──────────┐
          ▼ YES                ▼ NO
    Honeypot mode          Normal mode
    Gemini/OpenAI          Forward to real
    decoy response         upstream agent
          │
          ▼
    WebSocket Hub ──► Dashboard (Frontend)
```

### Services

| Service | Port | Role |
|---------|------|------|
| **backend** | 8081 | REST API, business logic, WebSocket streaming |
| **lobstertrap** | 8080 | Deep packet inspection, intent analysis |
| **redis** | 6379 | Storage for sessions, attacks, rules, personas, stats |

### Architectural advantages

**1. Deception-first approach**
Instead of blocking (which the attacker immediately notices) — deception. The attacker continues the attack, revealing techniques and objectives, while you collect IOC indicators.

**2. Separation of inspection and response**
Lobster Trap is only responsible for traffic analysis — it knows nothing about Redis or Gemini. The backend decides whether to switch modes. Any component can be replaced or updated independently.

**3. Graceful degradation**
If Lobster Trap is unavailable → backend falls back to a keyword-based detector. If the Gemini API is unavailable → static fallback responses are used. The system works even under partial failure.

**4. Real-time WebSocket streaming**
All events (new session, attack, status change) are immediately broadcast to the dashboard via the WebSocket hub. No polling, no delay.

**5. Rules engine with priorities**
Each rule has a `priority` (int). When multiple rules match the same technique, the rule with the highest priority wins. Supported actions: `LOG`, `QUARANTINE`, `ALLOW`.

**6. Per-session profiling**
Each session accumulates an `attacker_profile`: techniques used, intent category (adversarial / credential_access / data_exfiltration / social_engineering), risk_score. This allows multi-turn attacks (`multi_turn`) to be correlated.

**7. Demo mode**
The simulator generates realistic attacks every ~15 seconds — useful for demonstrations and rule testing without real attackers.

### Overview

**KPI cards at the top** — data for the last 24 hours, updated every 30 seconds:

|------|--------------|
| **Attacks caught · 24H** | Number of attacks detected in the last 24 hours. `↑ N%` — change vs. the previous period |

- Each bar = 1 time bucket

**Technique mix** — donut chart:
- Shows the distribution of attacks by technique type over 24 hours
- `TOTAL N` in the centre — total event count
- Legend on the right: each row = technique + attempt count

**Most-targeted agents** — table of agents hit most often:
- Sessions / Attacks / Decoy % / Health — per-agent statistics

**Attack origin** — geo-list of attack sources with indicator bars

### Settings

#### Threat Response

| Setting | Description |
|---------|------------|
| **Quarantine mode** | `strict` — full session isolation / `soft` — session continues with logging / `silent` — logging only, no behaviour change |
| **Auto-ban IP threshold** | Number of honeypot sessions from one IP within 24 hours before that IP is automatically blocked |
| **Trap depth ceiling** | Maximum decoy persona complexity: L1 (simple responses) → L2 → L3 (Persona + tool simulation) → L4 (Multi-agent decoy, most convincing) |
| **Require MFA for trap overrides** | Require a second factor for manual "Burn trap" and "Terminate decoy" actions |

#### Data & Retention

| Setting | Description |
|---------|------------|
| **Edge data retention** | How long to keep captured sessions and IOCs: 1 day / 7 days / 30 days / 90 days |
| **Decoy fixture rotation** | How often to automatically refresh fabricated datasets (1h / 24h / 7 days / Never). Prevents attackers from fingerprinting the trap by data structure |
| **Egress allowlist** | Whitelist of domains that outbound traffic is allowed to reach (for SIEM forwarding). By default nothing leaves the node |

#### System

| Setting | Description |
|---------|------------|
| **Auto-update signatures** | Automatically update the attack signature database |
| **Share anonymous telemetry** | Send anonymised statistics (disabled by default) |
| **Panic mode** | Emergency button — immediately put all traffic into quarantine |
| **Wipe all data** | Irreversibly delete all sessions, attacks, and logs |
| **Reset cluster** | Reset all settings to factory defaults |

## API: key endpoints

### Public (for agents)

```
POST /chat
Body:     { "session_id": "...", "agent_id": "...", "message": "..." }
Response: { "session_id": "...", "response": "...", "is_honeypot": true }
Rate limit: 60 requests/min per IP

POST /v1/chat/completions   (OpenAI-compatible drop-in)
Rate limit: 60 requests/min per IP

GET /health   (no auth required)
```

|-------|------|
